# xv6 전체 구조 분석

---

## 1. 부팅/초기화 흐름

```
entry.S → start.c → main.c
```

| 파일 | 역할 |
|------|------|
| `entry.S` | 가장 먼저 실행되는 어셈블리, CPU 초기 진입점 |
| `start.c` | machine mode에서 supervisor mode로 전환, 기본 환경 세팅 |
| `main.c` | 커널 진입점, 각 서브시스템을 순서대로 초기화 |

---

## 2. 핵심 4대 영역

### 2-1. 프로세스 (`proc.h`, `proc.c`)

**구성 요소**
- `proc.h` — 프로세스 상태 구조체 정의 (UNUSED, SLEEPING, RUNNABLE, RUNNING, ZOMBIE)
- `proc.c` — fork, exit, wait, scheduler 구현

**특징**
- 스케줄러: Round Robin 고정 (타임슬라이스 균등 분배)
- fork() — 부모 프로세스를 그대로 복사해 자식 생성
- exit() — 프로세스 종료 후 ZOMBIE 상태로 대기
- wait() — 부모가 자식 ZOMBIE 회수
- 멀티스레드 미지원 (프로세스 단위만 존재)

**장점**
- 동작이 예측 가능 → LLM 생성 코드 검증 용이
- 코드 구조 단순 → 수정 시 side effect 최소
- 기여도 측정 명확 (없는 기능을 추가하므로)

**단점**
- 우선순위/선점 스케줄링 없음 → 중요 프로세스 우대 불가
- 멀티스레드 미지원 → 병렬 처리 불가
- CPU 활용 효율 낮음 (모든 프로세스 동등 취급)

---

### 2-2. 메모리 (`kalloc.c`, `vm.c`, `memlayout.h`)

**구성 요소**
- `kalloc.c` — 물리 메모리 할당/해제 (free list 방식)
- `vm.c` — 가상 메모리, 페이지 테이블 관리
- `memlayout.h` — 메모리 주소 레이아웃 상수 정의

**특징**
- 물리 메모리를 4KB 페이지 단위로 관리
- free list: 빈 페이지를 연결 리스트로 유지
- Swap 없음 → 메모리 부족 시 kalloc() 단순 실패 반환
- 메모리 쿼터 없음 → 선착순 독점 가능
- OOM(Out of Memory) 킬러 없음 → 프로세스 그냥 죽음
- RR이 CPU는 균등 분배하지만 메모리는 균등 분배 보장 안 함

**장점**
- 구조 단순 → 이해 및 수정 용이
- 페이지 테이블 구조 명확 → 학습 목적에 최적
- 수정 시 side effect 최소

**단점**
- Swap 없음 → 물리 메모리 초과 시 처리 불가
- 메모리 단편화 처리 없음
- OOM 처리 없음 → 안정성 낮음
- 메모리 공정성 메커니즘 없음

---

### 2-3. 시스템 콜 (`syscall.h/c`, `sysproc.c`, `sysfile.c`, `usys.pl`)

**구성 요소**
- `syscall.h` — syscall 번호 정의
- `syscall.c` — 유저 → 커널 진입 처리, syscall 디스패치
- `sysproc.c` — 프로세스 관련 syscall 구현 (fork, exit, wait, kill 등)
- `sysfile.c` — 파일 관련 syscall 구현 (open, read, write, close 등)
- `usys.pl` — 유저 공간 syscall 래퍼 자동 생성 스크립트

**특징**
- 총 21개 syscall (read, write, fork, exec, exit, wait 등 기본만 존재)
- 유저가 ecall 명령 실행 → trap 발생 → syscall.c 디스패치
- 패턴 반복적: 번호 등록 → 핸들러 함수 작성 → usys.pl 추가

**장점**
- 구조 패턴 반복 → 자동 생성 최적화 포인트
- syscall 추가 시 side effect 적음
- 검증 기준 명확 (입력/출력 동작 확인)

**단점**
- 21개뿐 → 기능 극히 제한적
- 네트워크/스레드 관련 syscall 없음
- 복잡한 syscall 추가 시 코드 폭발 위험

---

### 2-4. 파일시스템 (`fs.h/c`, `file.h/c`, `bio.c`, `log.c`, `virtio_disk.c`)

**구성 요소**
- `bio.c` — 디스크 블록 캐시 (buffer cache)
- `log.c` — 로그 기반 간단한 크래시 복구
- `fs.c` — 파일시스템 핵심 로직 (inode, 디렉터리)
- `file.c` — 파일 디스크립터 관리
- `virtio_disk.c` — 가상 디스크 드라이버

**특징**
- 계층 구조: `disk → bio.c → log.c → fs.c → file.c`
- 저널링 없음 → 크래시 시 데이터 손상 가능
- log.c는 간단한 write-ahead log만 구현 (완전한 저널링 아님)
- 성능 최적화 없음 (캐시 교체 정책 단순)

**장점**
- 계층 구조 명확 → 흐름 추적 용이
- 각 계층 역할이 분리되어 있어 이해 쉬움

**단점**
- 저널링 없음 → 안정성 낮음
- 성능 최적화 전무
- 4대 영역 중 코드량 가장 많음 → LLM 적용 난이도 높음

**LLM 적용 방향**
- 파일시스템 계층별 코드 자동 설명 생성
- 저널링 추가 코드 제안
- bio.c 캐시 로직 버그 분석

## 3. 기타 영역

| 영역 | 파일 | 역할 |
|------|------|------|
| 트랩/인터럽트 | `trampoline.S`, `trap.c` | 유저↔커널 전환, 인터럽트 처리 |
| 동기화 | `spinlock.h/c`, `sleeplock.h/c` | 스핀락(busy-wait), 슬립락(대기) |
| 유저 공간 | `user.h`, `ulib.c`, `sh.c` | 유저 라이브러리, 쉘 구현 |

---

## 4. 파일 간 의존 흐름

```
main.c
  ├── kalloc.c / vm.c / proc.c
  │     └── syscall.c
  │           └── sysproc.c / sysfile.c
  │                 └── fs.c
  │                       └── bio.c
  └── trap.c
        └── trampoline.S
```
