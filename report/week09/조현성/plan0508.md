## 1. 시스템 개요

```
+-----------------------------------------------+
|  Host (macOS/Linux)                           |
|                                               |
|   +--------+   stdin/stdout pipe   +--------+ |
|   | QEMU   | <===================> | bridge | |
|   | + xv6  |                       |  .py   | |
|   +--------+                       +---+----+ |
|       ^                                |      |
|       |                                v      |
|   xv6 user space                  Upstage     |
|   - llmsh (자연어 shell)           Solar Pro 3|
|   - osdoc (자연어 → 진단)          (OpenAI    |
|   - schedhint (정책 제안)           compat)   |
|       |                                       |
|   xv6 kernel (우리가 직접 수정)               |
|   - setpriority/getpriority                   |
|   - sys_trace (syscall tracing)               |
|   - sys_sysinfo (proc/mem 통계)               |
|   - 우선순위 스케줄러 (이미 존재)             |
+-----------------------------------------------+
```

핵심 관점:
- **xv6 측**: 우리가 직접 구현한 시스템콜·스케줄러·동기화가 작동
- **Bridge 측**: QEMU의 콘솔 직렬 입출력을 호스트 파이썬 프로세스가 가로채 LLM과 통신
- **LLM 측**: Upstage Solar Pro 3가 자연어를 구조화 응답(JSON)으로 변환

## 2. xv6-riscv 현재 상태 점검

`os/2026-os/xv6-riscv/`를 실제로 열어보면 다음이 이미 들어와 있다.

| 영역 | 이미 구현됨 | 위치 |
|------|------------|------|
| 우선순위 스케줄러 | O — `proc.priority` 필드, single-pass best-so-far 선택 | `kernel/proc.c:432-479`, `kernel/proc.h:107` |
| `setpriority` / `getpriority` 시스템콜 | O | `kernel/syscall.c:104-105, 131-132`, `kernel/sysproc.c` |
| 우선순위 테스트 유저 프로그램 | O | `user/priority_test.c` |
| sysinfo / trace 관련 헤더 | O — `sysinfo.h`, `_sysinfo_test`, `_trace_test` 빌드 타깃 존재 | `kernel/sysinfo.h`, `user/_sysinfo_test`, `user/_trace_test` |
| 로그/스트레스/orphan 테스트 | O | `user/_logstress`, `_forphan`, `_dorphan`, `_spin` |

**의미**: 우선순위 스케줄러 자체는 이미 동작한다. 따라서 **LLM이 "무엇을 위해" 우선순위를 조절하느냐**가
프로젝트의 진짜 OS-기여 포인트가 된다. 단순히 priority를 바꾸는 wrapper만 만들면 §2 thin-wrapper 금지 조항에 걸린다.

## 3. 세 가지 서브시스템 (모두 같은 레포에서 통합)

### 3.1 `llmsh` — 자연어 Shell (조현성 안)

**위치**: `user/llmsh.c` (신규), `Makefile`의 `UPROGS`에 `$U/_llmsh` 추가.

**흐름**:
1. 프롬프트 표시 → 사용자가 한 줄 자연어 입력
2. `llmsh`가 해당 문자열을 콘솔로 그대로 출력 (`prefix` 토큰 포함, 예: `<<LLM>> ...`)
3. 호스트 측 bridge가 해당 라인을 캡처해 Solar API 호출
4. LLM은 **화이트리스트된 명령어 + 인자**만 JSON으로 반환
5. bridge는 응답을 다시 콘솔 stdin으로 보냄 (예: `<<EXEC>> ls /usr`)
6. `llmsh`가 EXEC 라인을 받으면 `fork() + exec()`으로 실행

**OS 개념**:
- `fork`/`exec`/`wait` 시스템콜 직접 사용
- 콘솔 I/O, 파이프 (LLM 자식 명령을 더 큰 파이프라인으로 묶을 때)
- **권한/위험 명령 화이트리스트**: 위험한 명령(`kill -9 1`, 미구현 syscall 등) 차단 로직을 user 측 + kernel 측에 이중 배치

**화이트리스트 (1차)**: `ls, cat, echo, grep, wc, mkdir, rm, ln, sysinfo_test, trace_test, priority_test`

**LLM 출력 스키마 (강제)**:
```json
{ "argv": ["ls", "/"], "rationale": "사용자가 루트 디렉터리 목록을 원함" }
```
스키마에서 벗어나는 응답은 bridge 단계에서 reject. `llmsh`는 항상 argv 배열만 신뢰.

### 3.2 `schedhint` — LLM 스케줄러 힌트 오라클 (조현성 안 + 박규태 분석)

xv6 우선순위 스케줄러는 이미 있다. 우리가 추가하는 건:

- **kernel 측**: `sys_proclist` 시스템콜 신설 — 모든 RUNNABLE/RUNNING 프로세스의 `pid, name, priority, sleeping_ticks, run_ticks` 스냅샷 반환
  - `proc.h`의 `struct proc`에 `uint64 run_ticks; uint64 sleep_ticks;` 추가
  - `proc.c`의 `scheduler()`와 `sleep()/wakeup()`에서 카운터 업데이트
  - Project.md §2 OS 개념: **scheduling, synchronization (proc table 락 필요)**
- **user 측**: `schedhint.c`가 주기적으로 `proclist` 스냅샷을 콘솔에 dump
- **bridge/LLM**: dump 분석 → "지금 spin은 우선순위 18로, sh는 5로" 식의 **힌트** 반환
- **user 측**: 받은 힌트를 `setpriority(pid, p)`로 적용

**핵심 안전장치 (박규태 보고서 + 한계 분석)**:
- LLM 제안은 **권고(hint)**일 뿐. kernel 측에서 다음 가드를 강제:
  1. `priority < 1`은 `init`/`sh` 외 거부 (starvation 방지)
  2. 한 번에 변경되는 priority delta ≤ 5
  3. 같은 pid에 대한 setpriority 호출 빈도 제한 (rate limit)
- 평가 지표: spin/IO 혼합 워크로드에서 **평균 응답시간**과 **starved process count**를 LLM-on/off로 비교

---

### 3.3 `osdoc` — 자연어 진단 (Kgent 안의 축소판, 이찬주 안)

eBPF는 xv6에 없다. 대신 우리가 직접 구현한 **kernel-level tracepoint**에 자연어 질의를 매핑한다.

- **kernel 측**: `sys_trace(int mask)` (이미 `_trace_test` 빌드 타깃이 존재 → 부분 구현 추정)
  - `proc->trace_mask` 필드 추가
  - `syscall.c`의 `syscall()`에서 mask 비트가 켜진 syscall 호출 시 `printf("[trace] pid=%d %s -> %d\n", ...)` 출력
- **user 측**: `osdoc.c`가 자연어 질의를 받음 ("최근 fork가 몇 번 호출됐는지 보고 싶어")
- **bridge/LLM**: 질의 → 적절한 `trace_mask` 비트 + 후처리 명령 결정
- **user 측**: 받은 mask로 `trace()` 호출 후 일정 시간 동안 dmesg 캡처 → bridge가 LLM에 다시 보내 "사람 말로 요약" 받기

**OS 개념**:
- **시스템콜 추가 (trace)**, **프로세스별 상태 (trace_mask 상속/격리)**, **로그 출력**
- Kgent 보고서의 Generate→Verify→Score 파이프라인 중 **Verify 부분을 mask 검증으로 대치** (잘못된 비트 거부)

## 4. 통합 데이터 흐름 예시

사용자 입력: `"프로세스 너무 많이 도는 것 같아, 누가 CPU 잡아먹는지 보고 우선순위 좀 깎아줘"`

```
llmsh: 자연어 입력 캡처
   ↓
bridge: Solar API 호출 (의도 분류: "diagnose + adjust")
   ↓
LLM 응답: { "plan": [
   { "tool": "osdoc", "args": ["top-cpu"] },
   { "tool": "schedhint", "args": ["lower", "spin", 5] }
]}
   ↓
llmsh: 단계별 실행
   - osdoc top-cpu  → sys_proclist() → spin이 1위 출력
   - schedhint lower spin 5 → setpriority(spin_pid, current+5)
   ↓
사용자에게 결과 요약 표시
```

## 8. 평가 지표 (Week 12 정의 의무)

1. **자연어 정확도**: 화이트리스트 명령 30개 시나리오에서 LLM이 올바른 argv를 반환한 비율

## 9. 리스크 & 미해결 결정

- **R1. QEMU 콘솔만으로 양방향 통신이 충분한가** — 가상 시리얼이 한 채널이라 LLM 입출력과 사용자 입출력이 섞일 수 있음. 대안: bridge 측에서 prefix 토큰(`<<LLM>>`, `<<EXEC>>`)으로 분리 → **1차 채택**.
- **R2. Solar API 응답 지연** — 매 명령마다 LLM을 부르면 shell 체감이 느림. 대안: 구조적인 명령(`ls`, `cd`)은 정규식 단축 경로로 우회.
- **R3. 안전장치 부족 시** — LLM이 root 권한 명령 제안 가능. xv6는 권한 모델이 단순(root only)이라, 화이트리스트가 유일한 선이다. 의도적으로 broad하지 않게 시작.
- **R4. xv6 부팅 시 LLM 의존** — `init`이 `llmsh`를 직접 호출하지 않게 해야 함. `sh`를 default로 두고, 사용자가 `llmsh`를 명시적으로 실행.
