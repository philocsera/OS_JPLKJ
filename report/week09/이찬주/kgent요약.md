# Kgent 기반 운영체제 개발 보조 프로그램 제안

## 1. 제안 배경

본 제안서는 xv6의 부족한 부분을 직접 보완하는 것을 목표로 하지 않는다.  
오히려 xv6가 실제 운영체제에 비해 단순하고 기능적으로 제한되어 있지만, 구조가 작고 명확하여 커널 수준의 실험을 비교적 쉽게 수행할 수 있다는 점에 주목한다.

즉, xv6는 완성된 운영체제를 대체하기 위한 대상이 아니라, 운영체제 내부 구조를 이해하고 새로운 OS 개발 보조 프로그램의 가능성을 검증하기 위한 실험 환경으로 사용된다.

---

## 2. 핵심 아이디어

이 프로젝트의 핵심 아이디어는 사용자가 자연어로 운영체제 관련 요구사항을 입력하면, 그 목적에 맞는 eBPF 코드를 생성하고 검증하는 OS 개발 보조 프로그램을 설계하는 것이다.

예를 들어 사용자는 다음과 같은 요구를 자연어로 입력할 수 있다.

```text
특정 시스템 콜의 호출 횟수를 추적하고 싶다.
프로세스별 CPU 사용 흐름을 관찰하고 싶다.
스케줄러 동작 중 특정 상태 변화를 기록하고 싶다.
메모리 할당이 발생하는 지점을 추적하고 싶다.
```

그러면 시스템은 입력된 요구사항을 분석하고, 해당 목적에 맞는 eBPF 코드 또는 eBPF-like 관찰 코드를 생성하는 방향으로 동작한다.

---

## 3. xv6를 실험 환경으로 사용하는 이유

xv6는 실제 Linux와 같은 복잡한 운영체제에 비해 기능이 제한적이다.  
하지만 바로 이 단순함 때문에 다음과 같은 장점을 가진다.

- 커널 구조가 작고 이해하기 쉽다.
- 프로세스, 스케줄러, 시스템 콜, 메모리 관리 흐름을 코드 수준에서 추적할 수 있다.
- 커널 코드를 수정하고 결과를 확인하는 실험이 비교적 쉽다.
- OS 개발 보조 프로그램의 아이디어를 빠르게 검증할 수 있다.
- 복잡한 실제 OS에 적용하기 전, 작은 환경에서 설계 가능성을 확인할 수 있다.

따라서 xv6는 기존 OS의 부족함을 해결하기 위한 최종 적용 대상이라기보다는, 자연어 기반 OS 코드 생성 및 검증 구조의 가능성을 실험하기 위한 기초 환경으로 적합하다.

---

## 4. 기존 Kgent 구조와 재설계 필요성

기존 Kgent는 자연어 요구사항을 기반으로 eBPF 코드를 생성하고, verifier를 통해 안전성을 확인하는 구조를 가진다.

기본적인 흐름은 다음과 같이 볼 수 있다.

```text
요구사항 입력
↓
eBPF 코드 생성
↓
Verifier 검증
↓
코드 출력
```

하지만 단순히 verifier를 통과한 코드가 항상 좋은 코드라고 볼 수는 없다.

운영체제 개발 관점에서는 다음과 같은 추가 판단이 필요하다.

- 생성된 코드가 실제 목적에 맞게 동작하는가?
- 성능 오버헤드는 적절한가?
- 커널 내부 상태를 안전하게 관찰하는가?
- 특정 상황에서 잘못된 결과를 만들 가능성은 없는가?
- 여러 코드 후보 중 어떤 코드가 더 효율적인가?
- 수정 후에도 verifier를 안정적으로 통과하는가?

따라서 본 제안서는 기존 Kgent의 pipeline 구조를 단순 코드 생성 중심에서 벗어나, 생성된 코드의 효율성과 실험 가능성을 함께 평가하는 구조로 재설계하고자 한다.

---

## 5. 제안하는 확장 Pipeline 구조

본 프로젝트에서는 기존 Kgent 구조를 다음과 같이 확장한다.

```text
1. Generate
   사용자의 자연어 요구사항을 기반으로 eBPF 코드 후보를 생성한다.

2. Verify
   Symbolic Verifier와 eBPF Verifier를 통해 hard constraint를 검사한다.

3. Checkpoint
   검증을 통과한 코드를 안전 후보로 저장한다.

4. Classify
   생성된 코드의 목적과 유형을 분류한다.
   예: 시스템 콜 추적, 스케줄러 관찰, 메모리 추적, 성능 측정 등

5. Score
   코드 유형에 따라 성능, 안정성, 효율성, 목적 적합성 등의 기준으로 점수화한다.

6. Optimize
   점수가 낮은 항목을 중심으로 제한된 코드 수정을 수행한다.

7. Re-verify
   수정된 코드가 다시 verifier를 통과하는지 확인한다.

8. Rollback or Accept
   검증에 실패하면 마지막 checkpoint로 되돌아가고,
   성공하면 새로운 안전 후보로 갱신한다.
```

이 구조는 단순히 코드를 생성하는 것에서 끝나지 않고, 생성된 코드가 실제로 목적에 맞고 효율적인지를 반복적으로 확인하는 데 초점을 둔다.

---

## 6. 제안 구조의 의미

이 제안의 목적은 xv6 자체를 완성도 높은 운영체제로 개선하는 것이 아니다.

핵심은 xv6의 단순한 구조를 이용해 다음 질문을 실험적으로 확인하는 것이다.

```text
자연어 요구사항을 기반으로 OS 관찰 코드를 생성할 수 있는가?
생성된 eBPF 코드가 verifier를 통과하는지 자동으로 검증할 수 있는가?
검증된 코드 중에서도 더 효율적인 코드를 선택할 수 있는가?
점수화와 재검증 구조를 통해 코드 품질을 반복적으로 개선할 수 있는가?
작은 OS 실험 환경에서 확인한 구조를 실제 운영체제 개발 보조 도구로 확장할 수 있는가?
```

따라서 이 프로젝트는 “xv6의 한계를 보완하는 프로젝트”라기보다는, xv6를 활용하여 OS 개발 보조 프로그램의 구조적 가능성을 검증하는 실험적 제안이다.

---

## 7. 기대 효과

본 프로젝트를 통해 기대할 수 있는 효과는 다음과 같다.

- 자연어 기반 eBPF 코드 생성 구조를 이해할 수 있다.
- Kgent의 기존 pipeline을 분석하고 확장할 수 있다.
- verifier 중심의 안전성 검증 구조를 실험할 수 있다.
- 단순 코드 생성이 아니라 성능과 효율을 함께 고려하는 평가 구조를 설계할 수 있다.
- xv6를 활용해 OS 내부 동작 관찰, 실험, 검증 과정을 체계적으로 정리할 수 있다.
- 향후 실제 Linux eBPF 환경으로 확장할 수 있는 기초 아이디어를 마련할 수 있다.
