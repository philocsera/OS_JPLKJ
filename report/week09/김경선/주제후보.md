# 프로젝트 주제 후보 정리

## 후보 A: 의미 기반 우선순위 결정

### 핵심 아이디어

프로세스 이름과 행동 패턴을 LLM에게 보내면, LLM이 "이 프로세스가 얼마나 중요한지" 판단해서 우선순위를 추천한다. 커널은 그 추천을 받아 `setpriority`를 호출한다.

### 왜 기존 방식으로 안 되나

기존 스케줄러는 프로세스의 **의미**를 모른다.

| 프로세스 | 실제 중요도 |
|----------|------------|
| `spin` (무한루프 테스트) | 낮음 |
| `sh` (사용자 셸) | 높음 |
| `cat` (파일 출력) | 중간 |

같은 우선순위로 시작하기 때문에 `spin`이 CPU를 다 잡아먹으면 `sh`가 버벅거린다.  
사람은 "spin은 테스트용이고 sh는 사용자가 쓰는 거니까 sh가 더 중요해"라고 판단할 수 있지만, 기존 스케줄러는 이런 판단을 못 한다.

### 동작 흐름

```
1. 사용자 프로그램(proclist)으로 현재 프로세스 목록 수집
2. 각 프로세스의 이름, CPU 사용시간, 메모리 사용량 등을 모음
3. bridge로 전송 (<<PROCS>> 형식)
4. LLM이 분석: "spin은 테스트, sh는 셸, cat은 명령어..."
5. LLM이 응답: 각 pid별 추천 우선순위 + 이유
6. xv6가 setpriority로 적용
```

### LLM에게 보내는 정보 예시

```
<<PROCS>>
pid  name      cpu_ticks  state
3    spin      8000       RUNNING
4    stressfs  5000       RUNNABLE
5    sh        200        SLEEPING
6    cat       50         RUNNABLE
<<PROCS_END>>
```

### LLM 응답 예시

```json
{
  "priorities": [
    {"pid": 3, "priority": 3,  "reason": "spin is infinite loop test"},
    {"pid": 4, "priority": 5,  "reason": "stressfs is stress test"},
    {"pid": 5, "priority": 15, "reason": "sh is user shell, needs response"},
    {"pid": 6, "priority": 12, "reason": "cat is user command"}
  ]
}
```

### 평가 방법

- 일반 xv6와 LLM 스케줄러 환경에서 같은 워크로드 실행
- `sh`의 명령 입력 → 응답까지 걸리는 시간 측정
- 차이가 크게 나야 "LLM 덕분에 좋아졌다" 주장 가능

### 구현해야 할 것

- [ ] 사용자 프로그램 (`prioagent.c`) — `proclist` 호출 후 `bridge`로 전송
- [ ] `bridge.py`에 `<<PROCS>>` 핸들러 추가
- [ ] LLM 프롬프트 작성 (우선순위 판단 가이드)
- [ ] LLM 응답 파싱 후 `setpriority` 호출

> **이미 있는 것:** `sys_proclist`, `setpriority`, `getpriority`, `bridge` 기본 구조

### 위험 요소

- 프로세스 이름이 너무 짧거나 의미 없으면 LLM이 판단 못 함
- LLM이 매번 다른 답을 주면 평가 어려움
- 우선순위 자주 바뀌면 시스템이 불안정해질 수 있음

## 후보 B: 의도 기반 워크로드 분류

### 핵심 아이디어

지금 시스템 전체가 어떤 상황인지를 LLM이 파악하고, 그 상황에 맞는 스케줄링 **정책**으로 자동 전환한다. 단순히 우선순위만 바꾸는 게 아니라 정책 자체를 바꾼다.

### 4가지 정책

| 정책 | 설명 | 적합한 상황 |
|------|------|------------|
| `throughput` | 타임슬라이스 길게 | 컴파일 같은 CPU 집중 작업 |
| `interactive` | 타임슬라이스 짧게 | 사용자 입력 응답성 우선 |
| `background` | 모든 우선순위 낮춤 | 백그라운드 작업 |
| `balanced` | 일반 라운드 로빈 | 기본 상태 |

### 왜 기존 방식으로 안 되나

리눅스 같은 OS는 CFS 같은 고급 스케줄러를 쓰지만, 그것도 "지금 사용자가 컴파일 중인지 게임 중인지" 추론하지는 못한다. 일반적인 휴리스틱(우선순위, vruntime)으로만 작동한다.  
LLM은 프로세스 이름과 행동을 종합해서 "지금이 어떤 상황인지" 추론할 수 있다.

### 동작 흐름

```
1. 몇 초마다 시스템 전체 스냅샷 수집
2. LLM에게 "현재 워크로드가 뭐 같아?" 질문
3. LLM 응답: workload 타입 + 추천 정책
4. 커널이 정책 전환
```

### LLM에게 보내는 정보 예시

```
<<WORKLOAD>>
process count: 5
processes:
  pid  name   cpu%  state
  3    gcc    85    RUNNING
  4    ld     70    RUNNABLE
  5    make   20    SLEEPING
  6    gcc    80    RUNNABLE
  7    sh     2     SLEEPING
<<WORKLOAD_END>>
```

### LLM 응답 예시

```json
{
  "workload": "compile",
  "policy": "throughput",
  "timeslice": 20,
  "reason": "Multiple gcc/ld processes, CPU intensive, compilation workload"
}
```

### 평가 방법

- 정해진 시나리오(컴파일, 대화형, 혼합)에서 실행
- LLM 끈 상태 vs 켠 상태 비교
- 컴파일 시간, 응답시간 측정

### 구현해야 할 것

- [ ] 정책 여러 개를 커널에 구현 (스케줄러 코드 수정)
- [ ] 동적으로 정책 전환하는 메커니즘
- [ ] 사용자 프로그램 (`workloadagent.c`) — 주기적으로 상태 수집
- [ ] `bridge.py`에 `<<WORKLOAD>>` 핸들러 추가
- [ ] LLM 프롬프트 (워크로드 분류 가이드)

### 위험 요소

- 정책 자체를 여러 개 만드는 게 큰 부담 (스케줄러 코드 수정)
- 정책 너무 자주 바꾸면 불안정
- 워크로드 분류가 애매한 경우 많음

## 후보 A 구현 계획: 의미 기반 우선순위 결정

### 1. 한 문단 요약 (Week 9 제출용)

xv6의 기존 스케줄러는 프로세스의 의미를 알지 못하기 때문에, 테스트용 무한루프 프로그램과 사용자 셸을 동일한 우선순위로 취급한다. 본 프로젝트는 LLM이 프로세스 이름과 CPU 사용 패턴을 분석하여 각 프로세스의 실제 중요도를 추론하고, 그 결과를 xv6의 `setpriority` 시스템 콜을 통해 실시간으로 적용하는 시스템을 구현한다. 사용자 셸(`sh`)처럼 응답성이 중요한 프로세스는 높은 우선순위를 얻고, `spin` 같은 테스트 프로그램은 낮은 우선순위로 자동 조정된다. 이를 통해 LLM이 OS 스케줄링 문제를 실질적으로 개선함을 정량 측정(응답 시간 비교)으로 증명한다.

### 3. 사용하는 OS 개념

| OS 개념 | 어디에 적용되나 |
|---------|---------------|
| 프로세스 상태 (`struct proc`) | `sys_proclist`로 전체 프로세스 정보 수집 |

| 라운드 로빈 + 우선순위 | 기존 스케줄러에 우선순위 가중치 통합 |

### 6. bridge.py 핸들러와 프로토콜

**프로토콜 형식**

```
<<PROCS>>
pid  name      cpu_ticks  state
3    spin      8000       RUNNING
4    stressfs  5000       RUNNABLE
5    sh        200        SLEEPING
6    cat       50         RUNNABLE
<<PROCS_END>>
```

**bridge.py 핸들러 추가 내용**

```python
if "<<PROCS>>" in line:
    collecting = True
    proc_lines = []
elif "<<PROCS_END>>" in line and collecting:
    collecting = False
    response = call_llm_for_priority(proc_lines)   # LLM 호출
    # JSON 응답을 그대로 prioagent의 stdin으로 전달
    # prioagent.c에서 직접 파싱 후 setpriority() 시스콜 호출
    send_to_xv6(response + "\n")
```

> **설계 원칙:** bridge.py는 JSON을 xv6 stdin으로 전달하고, `prioagent.c`가 JSON을 파싱해서 `setpriority()` 시스콜을 직접 호출한다. bridge.py가 셸 명령을 조립하는 방식은 쓰지 않는다.

You are an OS scheduling advisor for xv6, an educational operating system.

Guidelines:
- Shell processes (sh): priority 15-20 (user-facing, needs fast response)
- User commands (cat, echo, ls, grep): priority 10-14
- Background/stress tests (spin, stressfs, forktest): priority 1-5
- Build tools (make, gcc, ld): priority 6-10
- Unknown processes: priority 8 (neutral default)

Respond ONLY with a JSON object in this exact format:
{
  "priorities": [
    {"pid": <int>, "priority": <int 1-20>, "reason": "<short english reason>"}
  ]
}
Do not include any explanation outside the JSON.

### 10. 평가 방법 / 측정 지표 / 시연 시나리오

**측정 지표**

| 지표 | 측정 방법 |
|------|----------|
| `sh` 응답 시간 | 명령 입력 후 프롬프트 재표시까지 시간 (ms) |
| CPU 독점률 | `spin`이 전체 CPU의 몇 %를 가져가는지 |
| 우선순위 변화 로그 | LLM이 내린 결정과 이유 기록 |

**시연 시나리오**

```
[준비] prioagent를 백그라운드로 실행 (xv6에서 fork 후 부모는 sh로 복귀)
       → prioagent는 주기적으로 proclist 수집 및 LLM 호출 반복

1. [LLM OFF] spin 3개 실행 → sh에서 명령 입력 → 응답 지연 확인
2. [LLM ON]  spin 3개 실행 → prioagent 동작 → LLM이 spin 낮추고 sh 높임
3. sh에서 동일 명령 입력 → 응답 시간 개선 확인
4. LLM 결정 로그 화면에 출력 ("spin: priority 3, reason: infinite loop test")
```

> **백그라운드 실행 방법:** xv6 `sh`는 `&` 연산자를 지원하지 않으므로, `prioagent.c` 내부에서 `fork()` 후 부모 프로세스가 바로 `exit()`하고 자식이 루프를 도는 방식으로 구현한다.

## 후보 B 구현 계획: 의도 기반 워크로드 분류

### 1. 한 문단 요약 (Week 9 제출용)

xv6의 라운드 로빈 스케줄러는 컴파일 중인지, 사용자가 대화형으로 쓰는지 등 시스템의 현재 상황을 구분하지 못하고 항상 동일한 방식으로 동작한다. 본 프로젝트는 LLM이 현재 실행 중인 프로세스들의 종류와 행동 패턴을 분석하여 워크로드를 분류(compile/interactive/background/balanced)하고, 그 결과에 따라 xv6 커널이 스케줄링 정책(타임슬라이스 크기, 우선순위 가중치)을 자동으로 전환하는 시스템을 구현한다. 컴파일 워크로드에서는 처리량을 높이고, 대화형 워크로드에서는 응답성을 높이는 방식으로, LLM이 OS 수준의 정책 결정에 실질적으로 개입함을 시연한다.

### 5. 새로 추가할 시스템 콜

| 시스템 콜 | 설명 |
|----------|------|
| `setpolicy(int policy)` | 스케줄링 정책 전환 (0=balanced, 1=throughput, 2=interactive, 3=background) |
| `getpolicy(void)` | 현재 정책 번호 반환 |
| `proclist(char *buf, int size)` | 전체 프로세스 정보 수집 (이미 있음) |

**정책별 파라미터 예시**

| 정책 | 타임슬라이스 | 우선순위 기본값 | 설명 |
|------|------------|--------------|------|
| `balanced` | 5 ticks | 10 | 기본 라운드 로빈 |
| `throughput` | 20 ticks | 8 | CPU 집중 작업 최적화, 컨텍스트 스위칭 최소화 |
| `interactive` | 2 ticks | 15 | 응답성 우선, 빠른 순환 |
| `background` | 10 ticks | 3 | 배치 작업 중 사용자 방해 최소화 |

> **`background` 정책 의도:** 사용자 활동이 거의 없는 상태에서 백그라운드 배치 작업이 돌 때, 모든 프로세스를 낮은 우선순위로 설정하여 혹시라도 들어오는 사용자 입력에 우선권을 줄 수 있도록 여유를 남겨두는 것이 목적이다.

### 10. 평가 방법 / 측정 지표 / 시연 시나리오

**측정 지표**

| 지표 | 측정 방법 |
|------|----------|
| 컴파일 완료 시간 | `make` 실행 후 종료까지 시간 (sec) |
| `sh` 응답 시간 | 명령 입력 후 프롬프트 재표시까지 시간 (ms) |
| 정책 전환 로그 | LLM 결정 이유 + 전환 시점 기록 |

**시연 시나리오**

```
[준비] workloadagent를 백그라운드로 실행
       (prioagent와 동일하게 fork() 후 자식이 루프, 부모는 exit)

[시나리오 1: 컴파일 워크로드]
1. gcc 프로세스 다수 실행
2. [LLM OFF] balanced 정책 유지 → 컴파일 시간 측정
3. [LLM ON] workloadagent 동작 → LLM이 "compile" 분류 → throughput 정책 전환
4. 동일 컴파일 시간 재측정 → 개선 확인

[시나리오 2: 대화형 워크로드]
1. sh 중심 환경
2. [LLM OFF] balanced → sh 응답 시간 측정
3. [LLM ON] LLM이 "interactive" 분류 → interactive 정책 전환
4. sh 응답 시간 재측정 → 개선 확인

[시나리오 3: 정책 자동 전환 (핵심 시연)]
컴파일 완료 → 대화형으로 전환되는 과정에서
정책이 throughput → interactive로 자동 전환되는 것을 실시간 시연
```

> **백그라운드 실행 방법:** xv6 `sh`는 `&` 연산자를 지원하지 않으므로, `workloadagent.c` 내부에서 `fork()` 후 부모 프로세스가 바로 `exit()`하고 자식이 루프를 도는 방식으로 구현한다.
