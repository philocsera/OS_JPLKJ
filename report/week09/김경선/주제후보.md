# 프로젝트 주제 후보 정리

---

## 후보 A: 의미 기반 우선순위 결정

### 핵심 아이디어

프로세스 이름과 행동 패턴을 AI에게 보내면, AI가 "이 프로세스가 얼마나 중요한지" 판단해서 우선순위를 추천한다. 커널은 그 추천을 받아 `setpriority`를 호출한다.

### 왜 기존 방식으로 안 되나

기존 스케줄러는 프로세스의 **의미**를 모른다.

| 프로세스 | 실제 중요도 |
|----------|------------|
| `spin` (무한루프 테스트) | 낮음 |
| `sh` (사용자 셸) | 높음 |
| `cat` (파일 출력) | 중간 |

같은 우선순위로 시작하기 때문에 `spin`이 CPU를 다 잡아먹으면 `sh`가 버벅거린다.  
사람은 "spin은 테스트용이고 sh는 사용자가 쓰는 거니까 sh가 더 중요해"라고 판단할 수 있지만, 기존 스케줄러는 이런 판단을 못 한다.

### 동작 흐름

```
1. 사용자 프로그램(proclist)으로 현재 프로세스 목록 수집
2. 각 프로세스의 이름, CPU 사용시간, 메모리 사용량 등을 모음
3. bridge로 전송 (<<PROCS>> 형식)
4. AI가 분석: "spin은 테스트, sh는 셸, cat은 명령어..."
5. AI가 응답: 각 pid별 추천 우선순위 + 이유
6. xv6가 setpriority로 적용
```

### AI에게 보내는 정보 예시

```
<<PROCS>>
pid  name      cpu_ticks  state
3    spin      8000       RUNNING
4    stressfs  5000       RUNNABLE
5    sh        200        SLEEPING
6    cat       50         RUNNABLE
<<PROCS_END>>
```

### AI 응답 예시

```json
{
  "priorities": [
    {"pid": 3, "priority": 3,  "reason": "spin is infinite loop test"},
    {"pid": 4, "priority": 5,  "reason": "stressfs is stress test"},
    {"pid": 5, "priority": 15, "reason": "sh is user shell, needs response"},
    {"pid": 6, "priority": 12, "reason": "cat is user command"}
  ]
}
```

### 평가 방법

- 일반 xv6와 AI 스케줄러 환경에서 같은 워크로드 실행
- `sh`의 명령 입력 → 응답까지 걸리는 시간 측정
- 차이가 크게 나야 "AI 덕분에 좋아졌다" 주장 가능

### 구현해야 할 것

- [ ] 사용자 프로그램 (`prioagent.c`) — `proclist` 호출 후 `bridge`로 전송
- [ ] `bridge.py`에 `<<PROCS>>` 핸들러 추가
- [ ] AI 프롬프트 작성 (우선순위 판단 가이드)
- [ ] AI 응답 파싱 후 `setpriority` 호출

> **이미 있는 것:** `sys_proclist`, `setpriority`, `getpriority`, `bridge` 기본 구조

### 위험 요소

- 프로세스 이름이 너무 짧거나 의미 없으면 AI가 판단 못 함
- AI가 매번 다른 답을 주면 평가 어려움
- 우선순위 자주 바뀌면 시스템이 불안정해질 수 있음

## 후보 B: 의도 기반 워크로드 분류

### 핵심 아이디어

지금 시스템 전체가 어떤 상황인지를 AI가 파악하고, 그 상황에 맞는 스케줄링 **정책**으로 자동 전환한다. 단순히 우선순위만 바꾸는 게 아니라 정책 자체를 바꾼다.

### 4가지 정책

| 정책 | 설명 | 적합한 상황 |
|------|------|------------|
| `throughput` | 타임슬라이스 길게 | 컴파일 같은 CPU 집중 작업 |
| `interactive` | 타임슬라이스 짧게 | 사용자 입력 응답성 우선 |
| `background` | 모든 우선순위 낮춤 | 백그라운드 작업 |
| `balanced` | 일반 라운드 로빈 | 기본 상태 |

### 왜 기존 방식으로 안 되나

리눅스 같은 OS는 CFS 같은 고급 스케줄러를 쓰지만, 그것도 "지금 사용자가 컴파일 중인지 게임 중인지" 추론하지는 못한다. 일반적인 휴리스틱(우선순위, vruntime)으로만 작동한다.  
AI는 프로세스 이름과 행동을 종합해서 "지금이 어떤 상황인지" 추론할 수 있다.

### 동작 흐름

```
1. 몇 초마다 시스템 전체 스냅샷 수집
2. AI에게 "현재 워크로드가 뭐 같아?" 질문
3. AI 응답: workload 타입 + 추천 정책
4. 커널이 정책 전환
```

### AI에게 보내는 정보 예시

```
<<WORKLOAD>>
process count: 5
fork_rate: 5/sec
io_rate: 2/sec
processes:
  pid  name   cpu%  state
  3    gcc    85    RUNNING
  4    ld     70    RUNNABLE
  5    make   20    SLEEPING
  6    gcc    80    RUNNABLE
  7    sh     2     SLEEPING
<<WORKLOAD_END>>
```

### AI 응답 예시

```json
{
  "workload": "compile",
  "policy": "throughput",
  "timeslice": 20,
  "reason": "Multiple gcc/ld processes, CPU intensive, compilation workload"
}
```

### 평가 방법

- 정해진 시나리오(컴파일, 대화형, 혼합)에서 실행
- AI 끈 상태 vs 켠 상태 비교
- 컴파일 시간, 응답시간 측정

### 구현해야 할 것

- [ ] 정책 여러 개를 커널에 구현 (스케줄러 코드 수정)
- [ ] 동적으로 정책 전환하는 메커니즘
- [ ] 사용자 프로그램 (`workloadagent.c`) — 주기적으로 상태 수집
- [ ] `bridge.py`에 `<<WORKLOAD>>` 핸들러 추가
- [ ] AI 프롬프트 (워크로드 분류 가이드)

### 위험 요소

- 정책 자체를 여러 개 만드는 게 큰 부담 (스케줄러 코드 수정)
- 정책 너무 자주 바꾸면 불안정
- 워크로드 분류가 애매한 경우 많음

## A vs B 비교 요약

| 항목 | 후보 A | 후보 B |
|------|--------|--------|
| **무엇을 결정하나** | 프로세스별 우선순위 (개별 판단) | 시스템 전체 정책 (전체 판단) |
| **커널 수정 범위** | 적음 (기존 `setpriority` 활용) | 많음 (정책 여러 개 새로 구현) |
| **AI 호출 시점** | 새 프로세스 생기거나 주기적 | 주기적 (5~10초마다) |
| **시연 방식** | "sh가 빨라졌다" 같은 개별 개선 측정 | "정책이 자동으로 전환된다" 보여주기 |
| **난이도** | ⭐⭐⭐ (중간) | ⭐⭐⭐⭐ (어려움) |
