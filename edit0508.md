# edit0508.md — xv6-copy 구현 보고서 (2026-05-08)

`OS_22/plan0508.md`에서 합의한 Direction B (LLM for OS) 설계를
`OS_22/xv6-copy/` 트리에 1차 구현했다. 이 문서는 어떤 파일을 어떻게 바꿨는지,
프로토콜이 어떻게 동작하는지, 빌드/실행 방법, 그리고 미해결 항목을 정리한다.

---

## 1. 작업 요약

원본은 `os/2026-os/xv6-riscv/`에 있고 우선순위 스케줄러까지만 들어 있었다.
`OS_22/xv6-copy/`로 트리 전체를 복사한 뒤, 같은 디렉터리 안에서
다음 세 축의 작업을 수행했다.

| 축 | 한 줄 요약 |
|----|-----------|
| 커널 — 관찰 | `trace_mask`, `run_ticks`, `sleep_ticks` 필드 + `sys_trace`, `sys_sysinfo`, `sys_proclist` 시스템콜 추가 |
| 사용자 공간 | `llmsh` (자연어 셸), `schedhint` (proclist 덤프 + setpriority 적용), `osdoc` (자연어 → 트레이스 마스크) 추가 |
| 호스트 사이드 | `bridge/bridge.py` + Solar Pro 3 시스템 프롬프트 3종 (`prompts/{llmsh,schedhint,osdoc}.txt`) |

LLM이 직접 커널을 건드리지 않는다는 plan0508.md §0의 원칙을 그대로 지켰다.
LLM은 항상 **호스트 측 프로세스**에서 동작하고, 커널 진입은 우리가 직접 만든
시스템콜만이 가능하다. 커널 측 가드 + bridge 화이트리스트 + `schedhint`의
delta clamp까지 3중 안전망을 깔았다.

## 2. 변경된 파일 (커널)

### 2.1 `kernel/proc.h`

- `enum procstate` 바로 뒤에 `struct proc_info` 추가.
  사용자 공간으로 복사해서 내보낼 proc 스냅샷 레이아웃이다.
  `_pad`를 두어 `uint64` 필드들이 8바이트 정렬되게 했다.
- `struct proc`에 세 필드 추가:
  ```c
  int trace_mask;       // syscall trace bitmask
  uint64 run_ticks;     // RUNNING 상태로 스케줄러에 dispatch된 횟수
  uint64 sleep_ticks;   // SLEEPING 상태로 관측된 스캔 횟수
  ```

### 2.2 `kernel/proc.c`

- `allocproc()` (≈ line 153): 신규 프로세스 생성 시 세 필드 0으로 초기화.
- `freeproc()` (≈ line 180): 슬롯 회수 시 세 필드 0으로 초기화.
- `kfork()` (≈ line 309): `trace_mask`는 부모로부터 상속, 카운터는 자식 0부터.
  → 한 번 trace 켜놓고 fork를 하면 자식까지 자동 추적되는 동작 (xv6 trace 패치 관행).
- `scheduler()` (≈ line 444 ~ 491):
  - 매 dispatch마다 `best->run_ticks++` (RUNNING으로 전환되는 시점 = 한 타임슬라이스 단위 ≈ 1 tick).
  - 같은 스캔 패스에서 마주친 `SLEEPING` proc은 `p->sleep_ticks++`.
    별도 walker를 안 두려고 의도적으로 산정 시점을 스케줄러로 합쳤다.
- 파일 후반에 두 헬퍼 추가:
  - `proc_count()` — `state != UNUSED`인 proc 수 (sysinfo용).
  - `proclist_fill(struct proc_info *out, int max)` — proc 테이블 스냅샷을
    커널 버퍼에 채워주는 함수. lock을 entry 단위로 잡고 풀어 race를 줄였다.

### 2.3 `kernel/kalloc.c`

- `freemem_count()` 추가 (kfree free list를 walk해 byte 수 반환).
  락은 `kmem.lock`을 정상적으로 잡고 푼다.

### 2.4 `kernel/defs.h`

- 상단 forward declaration에 `struct proc_info;` 추가.
- kalloc 영역에 `uint64 freemem_count(void);` 선언.
- proc 영역에 `uint64 proc_count(void);`,
  `int proclist_fill(struct proc_info *, int);` 선언.

### 2.5 `kernel/syscall.h`

기존 22, 23이 `setpriority`/`getpriority`로 점유되어 있으므로
24, 25, 26을 새 syscall 번호로 할당:

```c
#define SYS_trace       24
#define SYS_sysinfo     25
#define SYS_proclist    26
```

> 주의: `trace.patch`의 SYS_* 번호 (22, 23)와는 다르다. 우선순위 패치와
> 충돌하지 않도록 일부러 24~26으로 미뤘다.

### 2.6 `kernel/syscall.c`

- 세 함수에 대한 `extern uint64 sys_*(void);` 추가.
- `syscalls[]` 디스패치 테이블에 세 항목 추가.
- 새로 `static char *syscall_names[]` 테이블 추가 (트레이스 출력용).
- `syscall()` 본체에 트레이스 훅:
  ```c
  if(p->trace_mask & (1 << num)) {
    printf("<<TRACE>> pid=%d name=%s call=%s ret=%d\n", ...);
  }
  ```
  접두사 `<<TRACE>>`는 host bridge가 파싱하는 마커라서 포맷이 고정이다.

### 2.7 `kernel/sysproc.c`

상단에 `#include "sysinfo.h"`, `#define PROCLIST_MAX NPROC` 추가.
세 핸들러 추가:

- `sys_trace`: `argint(0, &mask); myproc()->trace_mask = mask; return 0;`
- `sys_sysinfo`: `freemem_count()` + `proc_count()`로 struct sysinfo 채워 `copyout`.
- `sys_proclist`:
  - 사용자 포인터 + max를 받음
  - 커널 정적 버퍼(`static struct proc_info buf[NPROC]`)에 `proclist_fill`로 채움
  - `copyout`으로 한 번에 사용자 공간으로 복사
  - 반환값은 실제로 채운 엔트리 개수

### 2.8 `user/user.h`

- 상단에 `struct sysinfo;` forward + `struct proc_info { ... }` 정의 (kernel/proc.h와 레이아웃 동일).
- 기존 시스템콜 prototype 뒤에 `trace`, `sysinfo`, `proclist` 추가.

### 2.9 `user/usys.pl`

`entry("trace");`, `entry("sysinfo");`, `entry("proclist");` 세 줄 추가.
Makefile이 빌드할 때 `usys.S` 어셈블리 stub을 자동 생성한다.

## 3. 사용자 공간 신규 프로그램

### 3.1 `user/llmsh.c`

자연어 셸. 동작 흐름:

1. `nl> ` 프롬프트 출력
2. 한 줄 입력 받음
3. `!cmd args...` 형태면 LLM 우회, 화이트리스트 체크 후 fork+exec (오프라인 테스트용)
4. 그 외에는 `<<LLM>> <text>` 형태로 stdout에 출력 → 호스트 bridge가 캡처
5. 다음 stdin 한 줄을 기다림. `<<EXEC>> argv0 args...`이면 화이트리스트 검사 후 실행, `<<DENY>> reason`이면 거부

화이트리스트 (변경 가능):
`ls cat echo grep wc mkdir rm ln sysinfo_test trace_test priority_test schedhint osdoc`

화이트리스트 검사는 **사용자 측 + bridge 측 양쪽**에 둬서, 한쪽이 깨져도 다른 쪽이 막는다.

### 3.2 `user/schedhint.c`

LLM 스케줄러 힌트 오라클의 사용자 측 절반.

- `proclist()`로 proc 테이블 스냅샷 획득.
- stdout으로 다음 포맷 출력:
  ```
  <<PROCS>>
  pid name state priority run_ticks sleep_ticks
  ...
  <<PROCS_END>>
  ```
- bridge가 `<<SETPRI>> <pid> <delta>` 또는 `<<NOOP>>`을 stdin에 돌려준다.
- 받은 delta를 `apply_hint()`로 적용:
  - delta는 [-5, +5]로 clamp
  - 결과 priority는 [0, 20]으로 clamp
  - **pid 1 (init), pid 2 (sh)는 priority 5 미만으로 못 내려가게**
  - 모든 LLM 권고에 대해 priority 1 미만 금지 (safety)
- 인자 두 개 (`schedhint <pid> <delta>`)로 호출하면 bridge 우회 모드.

### 3.3 `user/osdoc.c`

자연어 진단 도우미. 동작:

1. `osdoc <자연어 질문>` → stdout에 `<<DOC>> ...` 출력
2. bridge가 `<<TRACE_MASK>> 0xMASK <ticks>`로 응답
3. `trace(mask)` → `pause(ticks)` → `trace(0)`
4. 그 동안 커널의 `<<TRACE>>` 라인이 stdout에 흘러나오고 bridge가 모아 LLM에게 요약 의뢰
5. `<<DOC_END>>` 마커로 캡처 종료

`--mask 0x42 --ticks 50` 형태의 직접 모드 지원 (LLM 없이 테스트 가능).

### 3.4 `Makefile`

`UPROGS`에 5개 추가:
```
$U/_trace_test
$U/_sysinfo_test
$U/_llmsh
$U/_schedhint
$U/_osdoc
```

(앞의 두 개는 이미 소스가 있는데 UPROGS 등록만 빠져 있던 상태였다.)

## 4. 호스트 측 (bridge/)

```
bridge/
├── bridge.py          # 메인 프록시
├── prompts/
│   ├── llmsh.txt      # 자연어 → argv 변환 시스템 프롬프트
│   ├── schedhint.txt  # 스냅샷 → setpri 결정 프롬프트
│   └── osdoc.txt      # 자연어 → trace mask 변환 프롬프트
├── requirements.txt   # openai>=1.40.0
└── .env.example       # UPSTAGE_API_KEY 자리표시자
```

### bridge.py 구조

- `subprocess.Popen("make qemu-nox", ...)`로 QEMU+xv6 기동, stdio 직접 잡음
- 두 스레드:
  - 읽기 스레드: QEMU stdout을 라인 단위로 받아 `<<LLM>>`/`<<DOC>>`/`<<PROCS>>...<<PROCS_END>>` 마커를 가로채 LLM 호출
  - stdin 스레드: 사용자가 친 키스트로크는 그대로 QEMU에 전달 → 일반 `sh` 사용은 그대로 동작
- LLM 호출은 OpenAI SDK + Upstage `base_url=https://api.upstage.ai/v1`
  (Solar 모델은 OpenAI 호환이라 그대로 동작)
- 모든 LLM 응답은 **JSON 객체 강제** (`response_format={"type": "json_object"}`)
- 응답 검증 실패 시 → `<<DENY>>`/`<<NOOP>>`/`<<TRACE_MASK>> 0 0`로 변환

### 안전망 (bridge 측)

`safe_argv()`:
- argv는 list여야 하고 모두 str
- argv[0]은 `WHITELIST` 안에 있어야 함
- 어떤 인자도 `; & | < > $ \`` 메타문자를 포함하면 거부
  (xv6 `sh.c`는 quoting/escaping이 단순해서 의도치 않게 위험할 수 있음)

`handle_procs()`:
- `delta` 절대값이 5 초과면 noop
- pid ≤ 0이면 noop

`handle_doc()`:
- mask가 0이거나 음수이면 trace 무시 (커널에 0이 전달되면 trace는 그냥 비활성화됨)

### 오프라인 모드

`UPSTAGE_API_KEY`가 없거나 `--offline` 플래그가 켜지면 LLM을 호출하지 않고
`<<DENY>>`/`<<NOOP>>`만 돌려준다 → 빌드/CI 환경에서 회귀 테스트만 돌릴 때 유용.

## 5. 빌드 & 실행 (예정 절차)

> 본 보고서 시점(2026-05-08)에서는 호스트 머신이 macOS이고 RISC-V 크로스
> 컴파일러를 아직 깔지 않았기 때문에, 실제 `make qemu`까지 돌려서 검증한 단계는
> 아니다. 빌드 절차만 정리한다.

```bash
# 1. RISC-V 크로스 툴체인 (한 번만)
brew install riscv-software-src/riscv/riscv-tools qemu

# 2. xv6 빌드
cd OS_22/xv6-copy
make            # kernel/kernel + fs.img 산출

# 3. (옵션 A) 그냥 xv6 콘솔 띄워보기
make qemu       # 또는 make qemu-nox

# 4. (옵션 B) bridge를 통해 LLM 모드로 띄우기
cd bridge
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env  # UPSTAGE_API_KEY 채우기
export $(cat .env | xargs)
python3 bridge.py --qemu-cmd "make qemu-nox" --cwd ..
# QEMU가 뜨면 sh가 실행됨. `llmsh` 입력하면 자연어 셸 진입.
```

세 가지 시나리오 빠른 점검:

| 명령 | 기대 동작 |
|------|----------|
| `nl> 루트 디렉터리 보여줘` | bridge가 `<<EXEC>> ls /` 응답 → `ls /` 실행 |
| `! schedhint` (오프라인) | proclist 스냅샷 출력 후 `<<NOOP>>` 자체 처리 |
| `osdoc fork 호출 보고 싶어` | bridge가 `<<TRACE_MASK>> 0x2 30` 응답 → 30 tick간 fork 트레이스 |

## 7. 빠진 것 / 다음 할 일

1. **실제 빌드 검증 안 됨** — riscv-toolchain 깔린 머신(또는 도커)에서 `make` 한번 돌려야
   타입/링크 에러 잡힘. `clangd` LSP는 `-I.` 플래그를 모르므로 `kernel/types.h not found`
   같은 false positive를 잔뜩 띄웠다 (실제 GCC 빌드는 통과해야 함).
2. **Bridge ↔ QEMU 콘솔 인터리빙** — 사용자 stdin과 bridge 응답이 같은 채널을 공유한다.
   현재 prefix 토큰으로 분리하지만, `gets()`가 한 글자씩 읽어 race가 일어날 수 있다.
   확실한 해결은 `chardev`를 두 개로 분리한 QEMU 옵션 사용 (예: `-serial mon:stdio -serial pty`)
   — Week 12 안에 검토.
3. **schedhint의 평가 워크로드** — `spin × N + stressfs`로 starvation 측정하는
   스크립트가 아직 없다. `test-xv6.py`에 시나리오 추가 필요.
4. **Trace 출력 양** — `<<TRACE>>` 라인이 폭주하면 console buffer가 넘쳐 xv6가 멈춘다.
   `osdoc`이 `pause(ticks)`로 시간을 자르고 있어 실제로는 안전하지만, 운영상
   "최대 N 라인" 가드를 커널 측에 둘지 검토.
5. **LLM JSON 강제의 한계** — `response_format`이 모든 Solar 모델에서 동일하게 동작하는지
   확인 필요. 안 되면 클라이언트 사이드에서 정규식 추출로 fallback.
6. **xv6 측 화이트리스트 = 13개** — 우리가 새로 만든 `schedhint`, `osdoc`은 LLM이
   직접 실행할 수 있게 화이트리스트에 들어가 있다. 의도된 권한이지만, "LLM이
   LLM 도구를 호출"하는 재귀가 가능해진다는 뜻 — 데모 시 주의.
